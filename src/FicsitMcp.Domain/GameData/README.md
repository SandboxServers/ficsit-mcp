# Game data layer

This folder is the canonical, install-free Satisfactory game-data layer. It parses the
game's `Docs.json` into an immutable in-memory model and exposes it via
`IGameDataService`.

## What ships here

- `game-data.v1.json` — the **versioned, checked-in snapshot** embedded into the
  `FicsitMcp.Domain` assembly. It lets the server answer item/recipe/building queries on
  a machine that has never had Satisfactory installed.
  - **Format:** plain UTF-8 JSON, compact, short property keys (see the `[JsonPropertyName]`
    attributes on the model records). Default values (0 power, `false`, null clearance,
    empty descriptions) are omitted to keep the file small (~277 KB).
  - **Game version:** Satisfactory Steam build **`23300430`** (1.x release branch). The
    build id comes from `steamapps/appmanifest_526870.acf` → `buildid`.

## The source `Docs.json` (NOT checked in)

The raw docs are ~10.6 MB per locale and are **not** committed. On a machine with the game
installed they live at:

```
<Steam>/steamapps/common/Satisfactory/CommunityResources/Docs/en-US.json
```

Two non-obvious facts the parser handles:

1. **The file is UTF-16 LE (with BOM).** Reading it as UTF-8 yields garbage. `DocsJsonParser`
   reads with `Encoding.Unicode`.
2. **It is an array of per-native-class groups**, each `{ "NativeClass": "...", "Classes": [...] }`.
   Recipes are under `FGRecipe`; items are spread across `FGItemDescriptor*` /
   `FGResourceDescriptor` / etc.; production buildings under `FGBuildableManufacturer*`,
   `FGBuildableResourceExtractor`, `FGBuildableGenerator*`.

Rate math is done once at parse time: per-minute = amount × 60 / duration. Liquid/gas
amounts in the raw docs are **millilitres** and are converted to cubic metres (÷1000)
before rates are computed.

## Config override (newer game versions)

Players on a game version newer than the shipped snapshot can point the server at their
local `Docs.json`:

```
FICSITMCP_GameData__DocsPath = C:\...\Satisfactory\CommunityResources\Docs\en-US.json
```

When set, the file is parsed live (UTF-16) and overrides the embedded snapshot. A
configured-but-broken path fails loudly at startup naming the path — it never silently
falls back to the snapshot (that would mask the misconfiguration and serve stale data).

## Refreshing the snapshot for a new game release

The snapshot is regenerated by a test in `FicsitMcp.Tests`:
`SnapshotGenerator.Regenerate` (driven by `SnapshotGeneratorTests`). It is **skipped on CI**
and whenever the live `Docs.json` is absent, so it never breaks the build.

To regenerate after a game update:

1. Confirm the new `buildid` in `steamapps/appmanifest_526870.acf`.
2. Bump `GameDataSnapshotLoader.SnapshotGameBuildId` (and `SnapshotVersion` if the model
   schema changed; if you bump the version, also rename the asset + `EmbeddedResourceName`).
3. Run the generator against the new docs, e.g.:

   ```sh
   set FICSITMCP_REGEN_DOCS=C:\...\CommunityResources\Docs\en-US.json
   dotnet test --filter "FullyQualifiedName~SnapshotGeneratorTests"
   ```

4. Review the diff of `game-data.v1.json`, run the full test suite (golden rate tests will
   catch any regression), and commit.
